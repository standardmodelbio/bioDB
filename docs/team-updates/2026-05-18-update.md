# Introducing bioDB — SMB's unified biomedical data layer (2026-05-18)

**Date:** 2026-05-18 · **Author:** Brian

## The 30-second pitch

[**bioDB**](https://github.com/bschilder/bioDB) is a new SMB Python library:
it's the data-acquisition layer for any work that needs biomedical knowledge.
Genes, diseases, drugs, variants, ontology terms, gene sets — pulled from 9
public sources through one uniform API.

The goal is *don't reinvent per-source clients*. Every source ships in two
modes: a **targeted query** (one record at a time, hits the live API) and a
**bulk download** (whole-corpus parquet / TSV / GMT / VCF / FASTA). Downstream
SMB packages already depend on it — [BioDocs](https://github.com/bschilder/BioDocs)
uses bioDB as its data-acquisition layer for the gene-knowledge-base build.

## Why this matters to SMB

| You want to … | With bioDB |
|---|---|
| Look up one gene, disease, drug, variant, or ontology term | One-line call hits the live upstream — `monarch.query_entity("HGNC:1100")`, `opentargets_graphql.query_drug("CHEMBL25")`, `ols.get_term("mondo", "MONDO:0004975")` |
| Query the Monarch knowledge graph (1.46 M nodes) for multi-hop / pattern questions | `monarch.query_cypher(...)` against the public Neo4j — no driver, no auth |
| Search any of ~280 OBO Foundry ontologies (Mondo, HPO, EFO, GO, ChEBI, …) without a local OWL file | `ols.search("alzheimer", ontology="mondo")` — uses the EBI Ontology Lookup Service |
| Build a downstream knowledge base | bioDB is the data-acquisition layer; BioDocs is the first consumer. The contract is stable enough for other packages to start depending on it |
| Catch upstream schema drift early | Every source has live integration tests in CI — they fail the day an upstream renames a field, not the day a user hits the broken call |
| Onboard a teammate to the library | One runnable Jupyter tutorial per data source, demonstrating both modes against the real API |

## What's in the library

| Data source | Targeted-query API | Bulk download |
|---|---|---|
| [Open Targets](https://platform.opentargets.org/) | GraphQL — target / disease / drug / variant | Parquet FTP — `list_datasets`, `get_dataset`, ensure_cached_shards |
| [Monarch Initiative](https://monarchinitiative.org/) | BioLink REST + Neo4j Cypher | Per-association TSV readers |
| OBO / OWL ontologies (Mondo, HPO, EFO, GO, …) | EBI OLS4 REST + owlready2 entity walk | OBO/OWL download + parse, N-hop keyword expansion, ontology similarity |
| [UniProt](https://www.uniprot.org/) | REST — query_protein, get_features, get_dbxrefs | FTP FASTA — Swiss-Prot + TrEMBL |
| [Harmonizome](https://maayanlab.cloud/Harmonizome/) | REST catalog | Bulk TSV / GMT |
| [ClinVar](https://www.ncbi.nlm.nih.gov/clinvar/) | NCBI E-utilities | VCF + CLNSIG simplification |
| [GWAS Atlas](https://atlas.ctglab.nl/) | Per-trait lookup over cached metadata | MAGMA p-value matrix |
| [gProfiler](https://biit.cs.ut.ee/gprofiler/) | REST `gost` enrichment | Combined per-organism GMT |
| [MSigDB](https://www.gsea-msigdb.org/gsea/msigdb/) | Per-set JSON | Per-collection GMT |

The [Sources table in the README](https://github.com/bschilder/bioDB#sources)
is the canonical map and stays in sync as new sources land.

## What landed today

- **All 5 in-progress source rows completed** ([PR #9](https://github.com/bschilder/bioDB/pull/9)) — Monarch BioLink REST, UniProt FTP FASTA, ClinVar NCBI E-utils, GWAS Atlas per-trait, MSigDB per-set JSON. Every source row is now ✅ on both columns.
- **Monarch KG over Cypher** ([PR #10](https://github.com/bschilder/bioDB/pull/10)) — direct access to the [public Monarch Neo4j](https://neo4j.monarchinitiative.org/) for graph-shaped questions the REST API can't express.
- **Per-source Jupyter tutorials** ([PR #11](https://github.com/bschilder/bioDB/pull/11)) — see [`tutorials/`](https://github.com/bschilder/bioDB/tree/main/tutorials). Authoring them surfaced two silent schema breakages in the Open Targets Drug query; both are now guarded by a live regression test.
- **OLS4 client for any OBO Foundry ontology** ([PR #12](https://github.com/bschilder/bioDB/pull/12)) — `biodb.ols` wraps the EBI Ontology Lookup Service so callers can search / walk a Mondo / HPO / EFO / GO term without downloading the OWL file. 20 tests (10 mocked + 10 live).

## What's next

- Worked-example tutorial that *joins* two sources — "all Mondo descendants of disease X, enriched with Open Targets evidence and Monarch phenotype associations" — to show off the unified-layer story.
- Smoke-test for the bulk-download path on at least one Open Targets parquet shard.
- Extend the schema-drift canary pattern (added today for Open Targets Drug) to every other GraphQL query type — Variant in particular has a sprawling type that's likely to drift.
- Convenience helpers on `monarch.query_cypher` for two common patterns: 2-hop pathway lookup and "all phenotypes for a gene".
- Stand up a `bioDB → BioDocs` example pipeline in the BioDocs repo, so the data-layer contract has a concrete reference consumer.
